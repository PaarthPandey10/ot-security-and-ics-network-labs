# 2.5 – Infrastructure Build Guide: STP/RSTP Loop Prevention & Error Recovery

> A visual narrative documenting the deployment of sub-second Layer 2 convergence and self-healing, hardened edge boundaries to protect the switch control plane from broadcast storms.

---

## Overview

This guide details the migration of the Cisco SF300 from legacy Spanning Tree to Rapid Spanning Tree Protocol (RSTP). It demonstrates how host-facing boundaries were hardened using Administrative Edge properties and BPDU Guard to aggressively drop rogue switch insertions. Finally, it covers the configuration of the switch's Error Recovery engine to automate the reactivation of ports tripped by security violations.

---

## Workflow Steps

### 1. Act I: RSTP Global Activation
Legacy Spanning Tree Protocol (STP) takes up to 50 seconds to transition a port from blocking to forwarding. To ensure modern network performance, we migrated the global switch fabric to RSTP (802.1w).

**STP Status & Global Settings:**
![](../../screenshots/rstp_Page_01.png)
*Navigating to `Spanning Tree > STP Status & Global Settings`, the STP Operation Mode was transitioned to `Rapid STP`[cite: 6]. The bridge priority was maintained at the default `32768`[cite: 6].*

### 2. Act II: Edge Port Provisioning & BPDU Guard
Host laptops do not generate Bridge Protocol Data Units (BPDUs). If a BPDU enters a host port, it indicates a user has plugged an unauthorized switch into the network, creating a massive risk of a Layer 2 loop and subsequent broadcast storm. We locked down the host boundaries (FE19 and FE24) to prevent this.

**Hardening the Ingress Boundary:**
![](../../screenshots/rstp_Page_03.png)
*Editing the STP Interface Settings for port FE19. `Edge Port` was Enabled to allow immediate forwarding, and `BPDU Guard` was Enabled to act as a physical tripwire against rogue switches[cite: 6].*

### 3. Act III: Hardware State Validation
With the configuration injected, we validated that the switch ASIC successfully applied the Edge flags and BPDU protections to our active VLAN ports.

**Interface Settings Verification:**
![](../../screenshots/rstp_Page_02.png)
![](../../screenshots/rstp_Page_04.png)
*The STP Interface Setting Table confirms the operational state. Both FE19 (Home) and FE24 (Work) successfully reflect `Enabled` for both Edge Port and BPDU Guard statuses, and both ports sit in a healthy `Forwarding` state[cite: 6].*

### 4. Act IV: Error Recovery Automation (The Self-Healing Loop)
When BPDU Guard trips, it places the port into an `err-disable` state, completely shutting down data transit. By default, this requires a network administrator to manually log in and reset the port. To ensure resilience, we automated this process.

**Configuring the Cooldown Timers:**
![](../../screenshots/rstp_Page_05.png)
*Navigating to `Port Management > Error Recovery Settings`. We enabled Automatic ErrDisable Recovery for both `Port Security` and `STP BPDU Guard`[cite: 6]. The recovery interval was set to `300` seconds (5 minutes)[cite: 6].* 

*Note: If a rogue switch triggers a port shutdown, the ASIC will wait 300 seconds and automatically attempt to re-enable the port[cite: 6]. If the rogue device has been unplugged, traffic will resume normally. If the device remains, the port will instantly trip and shut down again.*

## Technical Specifications

* **Core L3 Engine:** Cisco SF300-24P 
* **Target Boundaries:** Port FE19 | Port FE24[cite: 6]
* **STP Standard:** 802.1w (Rapid STP)[cite: 6]
* **Security Protocols:** Administrative Edge | BPDU Guard[cite: 6]
* **Automated Recovery Interval:** 300s (5 Minutes)[cite: 6]
