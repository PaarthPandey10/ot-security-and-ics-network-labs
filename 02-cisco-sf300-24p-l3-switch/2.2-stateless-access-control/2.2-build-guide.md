# 2.2 – Infrastructure Build Guide: L3 Security & Stateless Access Control (ACL)

> A visual narrative documenting the deployment of a Zero-Trust unidirectional isolation policy using stateless ASIC-level packet filtering.

---

## Overview

This lab documents the transition from an open L3 routing fabric to a segmented Zero-Trust environment. We defined a Unidirectional Isolation Policy between the Trusted Home network (VLAN 10) and the Untrusted Work network (VLAN 20), prohibiting initiation of traffic from the Work network to the Trusted network using stateless IPv4 Access Control Lists (ACLs). This guide details the resolution of stateless filtering "Default Action" traps and definitive hardware-level traffic drop validation.

---

## Workflow Steps

### 1. Act I: The Zero-Trust Policy Architecture
The primary objective was to instantiate a unidirectional gate. We created an IPv4-Based ACL named `isolate-work` in the switch control plane to house our restrictive rules.

**ACL Container Instantiation:**
![](../../screenshots/acls_Page_01.png)
*The `isolate-work` ACL container initialized in the switch registry. This serves as the parent object for all Access Control Entries (ACEs).*

### 2. Act II & III: The Stateless ACL Trap (The Smoking Gun)
With the container staged, we provisioned the Access Control Entry (ACE) to deny ICMP Type 8 (Echo Request) traffic originating from the `192.168.20.0/24` subnet.

**ACE Provisioning (Deny ICMP Type 8):**
![](../../screenshots/acls_Page_02.png)
*Defining the specific ACE: Priority 10, Action: Deny, Protocol: ICMP, Type: Echo Request. This rule surgicality targets initiation attempts from the Untrusted VLAN.*

**The Stateless ACL Trap:**
During binding, the interface was accidentally configured with a "Default Action: Deny Any." This revealed a critical architectural difference: unlike stateful firewalls (e.g., Windows Defender), Cisco ACLs do not track connection states. When the Trusted DP (VLAN 10) pinged the Untrusted WL (VLAN 20), the WL generated a legitimate Echo Reply, but the switch ASIC blackholed it because it didn't explicitly match a "Permit" rule and hit the "Deny Any" default.

**Binding the ACL to the Ingress Boundary (FE24) with 'Permit Any' logic:**
![](../../screenshots/acls_Page_03.png)
*The critical remediation: Adjusting the binding Default Action to "Permit Any" ensures that while ICMP Echo Requests are blocked, legitimate return traffic is allowed to traverse the ASIC.*

### 3. Act IV: Packet-Level State Validation (Hex Dump Forensics)
With the policy remediated, we executed final validation. We successfully verified that traffic initiation from the untrusted network is physically stopped by the switch hardware.

**Packet-Level Forensics (Wireshark Capture):**
![](../../screenshots/acls_Page_04.png)
*Packet forensic capture showing successful transit of Echo Replies (Type 0) vs. hardware-generated drops (Type 3, Code 0) when the WL attempts to initiate pings.*

* **Successful Request:** Packet `5157` shows the DP (192.168.10.150) sending an Echo Request. Packet `5158` shows the WL responding with an Echo Reply. The ASIC allowed this because the return path was permitted.
* **Unauthorized Drop:** Packet `5167` shows the ASIC responding with "Destination Unreachable (Network unreachable)" when the WL attempts a new ping initiation. This confirms the ACL is functioning as a wire-speed hardware gatekeeper.

## Technical Specifications

* **Core L3 Engine:** Cisco SF300-24P (ASIC Filtering: ENABLED)
* **Target Boundary:** VLAN 20 (Untrusted Work) | Port FE24
* **ACL Name:** `isolate-work`
* **Filtering Logic:** Priority 10 | Deny ICMP Type 8 | Default Action: Permit Any
